## RAG系统面试核心问题梳理

### 1. 完整梳理检索增强生成（RAG）的标准执行流程
RAG的标准执行流程主要分为三个阶段：**索引阶段、检索阶段、生成阶段**。
*   **索引阶段：**
    1.  **数据清洗**：对原始文档进行去噪、格式转换。
    2.  **切片**：将长文档切分为较小的语义单元，通常保留重叠部分以保持上下文连贯。
    3.  **向量化**：使用Embedding模型将文本切片转化为向量。
    4.  **存储**：将向量及元数据存入向量数据库。
*   **检索阶段：**
    1.  **查询预处理**：对用户Query进行纠错、改写或意图识别。
    2.  **向量化**：将用户Query转化为向量。
    3.  **相似度计算**：在向量库中检索与Query最相似的Top-K个切片。
    4.  **重排序（可选但推荐）**：使用重排序模型对检索结果进行精细打分，筛选出最相关的文档。
*   **生成阶段：**
    1.  **上下文构建**：将检索到的文档切片与用户Query组合，构建Prompt。
    2.  **模型推理**：大模型根据上下文生成回答。
    3.  **结果输出**：返回答案及引用来源。
### 2. 落地RAG系统时，通常会选用什么类型的向量/存储数据库，各自适用场景是什么？
在实际落地中，我们通常从以下三类中进行选择：
1.  **专用向量数据库（如 Milvus, Pinecone, Weaviate）：**
    *   **适用场景**：超大规模数据、高并发检索需求、对检索性能要求极高的场景。
    *   **特点**：专为向量设计，支持分布式扩展，检索速度极快，支持标量过滤与向量检索的结合。
2.  **扩展型关系/文档数据库（如 PostgreSQL/pgvector, Elasticsearch, MongoDB）：**
    *   **适用场景**：项目初期、数据量中等、已有成熟数据库基建的场景。
    *   **特点**：运维成本低，无需引入新组件，能结合原有数据的结构化查询能力，但超大规模下的性能略逊于专用库。
3.  **图数据库（如 Neo4j）：**
    *   **适用场景**：知识图谱RAG（GraphRAG），需要处理多跳关系、复杂推理的场景。
    *   **特点**：存储实体关系，能弥补向量检索在逻辑推理上的不足。
### 3. 针对用户的复杂推理类查询，系统会如何加工处理？深度追问：如何将自然语言文本转化为图谱的节点与边结构？
**针对复杂推理：**
单纯依靠向量检索（语义相似）难以解决多跳推理问题。我会采用 **GraphRAG** 或 **知识图谱增强** 的方式。首先识别查询中的实体，通过图谱检索关联实体和关系，结合图谱路径信息（如A->B->C）作为上下文，辅助模型进行逻辑推演。
**图谱构建方案（文本转节点与边）：**
将非结构化文本转化为图谱结构，通常采用 **"大模型抽取 + 图谱写入"** 的方案：
1.  **定义Schema**：预先定义本体，明确需要抽取的实体类型（如“人物”、“公司”）和关系类型（如“任职于”、“位于”）。
2.  **LLM信息抽取**：利用大模型（如GPT-4）配合Prompt Engineering，执行实体关系抽取任务。Prompt中明确指示输出结构（如JSON格式：`{head, relation, tail}`）。
3.  **结构化转换**：模型输出文本后，通过正则或解析库提取三元组。
4.  **图数据库写入**：
    *   **节点处理**：对实体名称进行归一化处理（消歧），如果节点已存在则更新属性，不存在则创建。
    *   **边处理**：在两个节点间建立关系，并存储关系的来源文档ID等元数据。
### 4. 从原始文本数据中，抽象提取实体、实体间关系的整体思路是什么？
整体思路分为四步：
1.  **数据预处理**：清洗数据，分句或分段，确保输入文本粒度适中。
2.  **Schema约束设计**：这是最关键的一步。不能无限制抽取，必须结合业务场景定义核心实体和关系（例如做金融RAG，核心是“公司”、“股票”、“高管”）。可采用固定Schema或基于大模型的动态Schema。
3.  **抽取策略**：
    *   **Prompt Engineering**：设计Few-shot提示词，引导模型按JSON格式输出。
    *   **微调模型**：如果通用模型效果不佳，会微调小参数模型（如GLiNER或专用NER模型）用于降低成本和提高速度。
4.  **后处理与融合**：解决实体共指消解问题（如“阿里”和“阿里巴巴”合并），并将新提取的知识与已有图谱进行合并，避免重复节点。
### 5. RAG全链路存在推理延迟问题，针对查询改写、向量检索等环节，有哪些优化手段？
针对全链路延迟，各环节优化手段如下：
*   **查询改写环节**：
    *   **异步流式处理**：如果涉及多轮对话改写，使用流式输出。
    *   **小模型替代**：改写任务通常不需要GPT-4级别模型，可用Llama-3-8B或专用小模型替代，大幅降低首字延迟。
    *   **查询缓存**：对高频相同或相似查询建立缓存，直接返回结果。
*   **向量检索环节**：
    *   **索引优化**：使用HNSW（导航小世界图）索引替代暴力搜索，在精度损失极小的情况下换取极速检索。
    *   **量化技术**：对向量进行INT8量化，减少内存占用和计算耗时。
    *   **Top-K调优**：在召回率和性能间平衡，适当减少检索数量。
*   **生成环节**：
    *   **流式输出**：必须开启Server-Sent Events (SSE)，让用户边生成边看，体感延迟更低。
    *   **投机采样**：在模型推理中使用投机采样技术加速生成。
### 6. 你项目内如何设定RAG上下文窗口长度？怎样统一管理大模型输入上下文？
**设定上下文窗口长度：**
并不是越长越好。通常我会设定一个动态阈值：
*   **上限控制**：根据目标模型的上下文限制（如8k或128k）减去输出预留Token和系统提示词Token，计算出可用空间。
*   **性能考量**：通常单次检索上下文控制在2000-4000 Token左右。过长的上下文会导致模型“中间迷失”问题且推理成本剧增。
**统一管理输入上下文：**
采用 **"递进式填充法"**：
1.  **系统指令**：最高优先级，始终保留。
2.  **对话历史**：采用滑动窗口或摘要压缩机制，保留最近的N轮对话或历史摘要。
3.  **检索内容**：根据剩余Token空间动态填充检索到的文档切片。
4.  **用户Query**：放在最靠近模型输入的位置，确保模型优先关注当前问题。
### 7. 筛选哪些文本素材存入RAG知识库？当输入上下文占模型总长度1/4时容易出现注意力分散，有什么优化方案？
**筛选素材原则：**
*   **高信息密度**：优先存储核心业务文档、操作手册、FAQ。
*   **时效性与准确性**：剔除过时、自相矛盾的数据。
*   **去重与清洗**：去除页眉页脚、导航栏等噪音数据。
**针对注意力分散（Lost in the Middle）的优化方案：**
当上下文过长导致模型注意力分散时，除了缩短上下文外，核心技术是 **重排序**：
1.  **检索后重排序**：不要直接把检索到的Top-K切片全部塞给模型。使用BGE-Reranker或Cohere Rerank对切片进行精准打分。
2.  **精选与重排**：只选取重排序后得分最高的Top-3或Top-5切片。
3.  **位置优化**：将最相关的切片放置在Prompt的**开头**或**结尾**（研究表明模型对这两个位置注意力最强）。
4.  **指令微调**：如果是私有化模型，可以对模型进行针对长文档问答的微调，增强其抗干扰能力。
### 8. 针对用户结构化偏好信息，如何设计存储方案？能否全覆盖遍历全部用户偏好分类？
**存储方案设计：**
采用 **“结构化字段 + 向量化双写”** 策略：
1.  **结构化存储**：将明确的偏好（如“语言偏好：中文”、“行业偏好：金融”）存入关系型数据库或用户画像宽表中，用于精确过滤。
2.  **向量化存储**：将用户的兴趣标签、历史行为摘要转化为用户画像向量，存入向量库。
**能否全覆盖遍历：**
**不能也不建议全覆盖。**
*   **原因**：用户偏好是动态且无限的，预定义的分类无法涵盖所有边缘情况。
*   **解决方案**：采用 **混合检索策略**。
    *   对于显性偏好（如已勾选的分类），通过结构化查询强制过滤。
    *   对于隐性偏好，通过向量相似度检索匹配。
    *   最终在召回阶段通过加权融合两种结果，既保证核心偏好满足，又具备泛化能力。
### 9. 迭代优化RAG系统后，通过哪些指标、实验方案验证整体效果提升？
验证效果需建立自动化评估体系，分为三个维度：
**评估指标：**
1.  **检索指标**：Recall@K（召回率）、MRR（平均倒数排名）、Precision@K。使用验证集（人工标注的Query与相关文档对）计算。
2.  **生成指标**：
    *   **真实性**：是否产生幻觉？（通过RAGAS框架评估）。
    *   **相关性**：回答是否切题？
    *   **一致性**：答案是否可由检索内容支撑？
3.  **端到端指标**：人工评估或使用强模型（如GPT-4）作为裁判进行打分。
**实验方案：**
1.  **构建金标准数据集**：准备100-200条典型Query及其标准答案和关联文档。
2.  **A/B Test**：新旧系统并行跑分，对比各项指标。
3.  **Bad Case分析**：重点分析得分下降或回答错误的案例，反哺优化切片策略或Prompt。
