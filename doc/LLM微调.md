# LLaMA Factory 微调参数详解笔记

---

## 1. `pure_bf16`

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 是否全程用 bf16 精度训练，跳过 fp32 梯度累积 |

### 通俗理解

训练时，模型参数有两种精度策略：

```
False（AMP 混合精度）：
  前向计算 → bf16（快、省显存）
  梯度累积 → fp32（精确、稳定）
  ✅ 兼顾速度和稳定性

True（纯 bf16）：
  前向计算 → bf16
  梯度累积 → bf16（也省了）
  ⚠️ 更快更省，但可能 loss 爆炸 / NaN
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 正常训练 | `False` ✅ | 默认最稳 |
| 显存差一点点就 OOM | `True` | 省掉 fp32 梯度的显存开销 |
| 训练出现 NaN / loss 震荡 | 改回 `False` | 纯 bf16 精度不够 |

> 📌 **结论：99% 的情况保持 `False`，除非你明确知道自己在做什么。**

## 2. `stage`

| 项目 | 内容 |
|------|------|
| **类型** | `Literal["pt", "sft", "rm", "ppo", "dpo", "kto"]` |
| **默认值** | `sft` |
| **一句话解释** | 你要做哪种类型的训练 |

### 通俗理解

```
pt   → 预训练：给模型"喂"大量原始文本，让它学会语言/领域知识
sft  → 监督微调：给模型"问答对"，让它学会按指令回答
rm   → 奖励模型：训练一个"打分器"，判断回答好不好
ppo  → 强化学习：用打分器指导模型生成更好的回答（RLHF）
dpo  → 直接偏好优化：跳过打分器，直接用"好回答 vs 坏回答"对齐
kto  → KTO 对齐：只需要标记"好/坏"，不需要成对对比
```

### 什么时候改？

| 你的目标 | 设置 | 说明 |
|----------|------|------|
| 让模型学会某个领域的知识（如医学文献） | `pt` | 数据格式：纯文本 |
| 让模型学会按指令对话 | `sft` ✅ | 数据格式：instruction + output |
| 让模型输出更符合人类偏好 | `dpo` | 数据格式：chosen + rejected 成对 |
| 只有"好/坏"标签，没有成对数据 | `kto` | 数据格式：prompt + response + label(0/1) |
| 做完整 RLHF 流程 | `rm` → `ppo` | 分两步走 |

> 📌 **结论：大多数人选 `sft`。有偏好数据选 `dpo`。**

## 3. `finetuning_type` ⭐ 最重要的参数

| 项目 | 内容 |
|------|------|
| **类型** | `Literal["lora", "freeze", "full"]` |
| **默认值** | `lora` |
| **一句话解释** | 用哪种方式微调模型 |

### 通俗理解

```
full（全参数微调）：
  模型所有参数都更新
  📦 显存：极大（7B 模型约需 60-80GB）
  🎯 效果：最好（上限最高）
  ⏱️ 速度：最慢

lora（低秩适配）：
  冻结原模型，只训练插入的小矩阵（rank 通常 8~64）
  📦 显存：小（7B 模型约需 16-24GB）
  🎯 效果：接近 full 的 90%+
  ⏱️ 速度：快

freeze（冻结微调）：
  冻结模型底层，只训练顶层若干层
  📦 显存：中等
  🎯 效果：介于 lora 和 full 之间
  ⏱️ 速度：中等
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 单卡 24GB（4090/3090） | `lora` ✅ | 唯一选择 |
| 单卡 40-80GB（A100） | `lora` | 性价比最高 |
| 多卡 8×A100 80GB | `full` | 追求极致效果 |
| 效果不够，想比 lora 好一点 | `freeze` | 折中方案 |
| 快速实验/调参 | `lora` | 训练快，迭代快 |

> 📌 **结论：默认 `lora` 就对了。只有资源充足且追求极致效果时才用 `full`。**

## 4. `use_llama_pro`

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 是否用 LLaMA Pro 方法：冻结原模型，只训练新插入的扩展块 |

### 通俗理解

```
False（默认）：
  正常微调，更新所有参数（或 LoRA 参数）
  
True（LLaMA Pro 模式）：
  原模型权重 → 完全冻结，不动
  在模型中插入新的"扩展块" → 只训练这些新块
  
  类比：
  原模型 = 一本已经写好的书（不动）
  扩展块 = 在书里插入新的空白页（只写这些页）
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 普通 SFT / LoRA 微调 | `False` ✅ | 不需要 |
| 想注入新知识但**不想破坏原模型能力** | `True` | 防灾难性遗忘 |
| 持续学习场景（先学 A 再学 B，不想忘 A） | `True` | 原模型冻结，新块学新知识 |

> 📌 **结论：99% 的情况保持 `False`。只有"持续学习/知识注入且怕遗忘"时才考虑 `True`。**

## 5. `use_adam_mini`

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 是否用 Adam-mini 优化器来省显存 |

### 通俗理解

```
False（标准 AdamW）：
  每个参数都维护 m（一阶矩）和 v（二阶矩）
  显存开销 = 参数量 × 2（m 和 v 各一份）
  
True（Adam-mini）：
  把参数分成若干块，每块共享一组 m 和 v
  显存开销大幅减少（可省 50%+ 优化器显存）
  
  类比：
  AdamW = 每个学生配一个专属辅导员
  Adam-mini = 一个辅导员管一个班（省资源，但关注度下降）
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| LoRA 微调 | `False` ✅ | LoRA 本身已省显存，不需要 |
| 全量微调 7B，显存勉强够 | `False` | 先试默认 |
| 全量微调 70B+，优化器显存爆了 | `True` | 省优化器显存 |
| 训练效果变差 | 改回 `False` | 精度有损 |

> 📌 **结论：LoRA 用户完全不用管。只有全量微调超大模型且显存不够时才考虑。**

## 6. `freeze_vision_tower`（多模态专用）

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `True` |
| **一句话解释** | 多模态模型训练时，是否冻结视觉编码器（如 ViT） |

### 通俗理解

```
多模态模型结构：
  [视觉塔 ViT] → [投影层 Projector] → [语言模型 LLM]
       ↑                ↑                    ↑
  freeze_vision_tower  freeze_multi_modal   由 finetuning_type 控制
                       _projector

freeze_vision_tower = True：
  视觉塔不动（它已经能很好地"看图"了）
  只调整后面的部分
  
freeze_vision_tower = False：
  视觉塔也一起训练（让它学会"看新的图"）
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 纯文本任务（不涉及图片） | `True` ✅ | 视觉塔用不到，冻了省显存 |
| 图文对话微调（通用场景） | `True` ✅ | ViT 已经够好，不需要动 |
| 特殊视觉领域（医学影像/遥感/工业检测） | `False` | ViT 需要适配新领域 |
| 显存紧张 | `True` | 冻结省显存 |

> 📌 **结论：默认 `True`。只有你的图片类型很特殊（ViT 预训练没见过的）才设 `False`。**

---

## 7. `freeze_multi_modal_projector`（多模态专用）

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `True` |
| **一句话解释** | 是否冻结视觉→语言的投影层 |

### 通俗理解

```
投影层的作用：
  把视觉塔输出的"图像特征"翻译成语言模型能理解的"语言特征"
  
  类比：翻译官
  视觉塔说中文 → [投影层/翻译官] → 语言模型听英文

freeze = True：翻译官不动
freeze = False：翻译官也重新学习
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 纯文本任务 | `True` ✅ | 用不到 |
| 想让模型更好地理解图文关系 | `False` | 让投影层重新对齐 |
| 配合 `train_mm_proj_only=True` | `False` | 只训练投影层 |

> 📌 **结论：一般保持 `True`。做图文对齐训练时设 `False`。**

---

## 8. `train_mm_proj_only`（多模态专用）

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 是否**只**训练投影层，其他全部冻结 |

### 通俗理解

```
True 时：
  视觉塔 → 冻结 🔒
  投影层 → 训练 🔥（唯一训练的部分）
  语言模型 → 冻结 🔒
  
  这是最轻量的多模态适配方式
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 纯文本任务 | `False` ✅ | 不涉及多模态 |
| 多模态轻量适配（数据少/显存少） | `True` | 只调投影层，最快最省 |
| 多模态深度微调 | `False` | 需要训练更多部分 |

> 📌 **结论：多模态场景下，数据量少或想快速验证时设 `True`。**

## 9. `compute_accuracy`

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 评估时是否计算 token 级别的预测准确率 |

### 通俗理解

```
False：评估时只看 loss（交叉熵）
True：评估时额外计算"模型预测的下一个 token 是否正确"的百分比

例如：
  真实输出："今天天气很好"
  模型预测："今天天气不错"
  token 准确率 = 4/5 = 80%
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 正式训练 | `False` ✅ | 计算准确率有额外开销 |
| 调试/观察模型是否"学会了" | `True` | 比 loss 更直观 |
| 对比不同 checkpoint | `True` | 准确率更容易比较 |

> 📌 **结论：调试阶段设 `True`，正式训练设 `False`（省时间）。**

## 10. `disable_shuffling`

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 是否禁止打乱训练数据的顺序 |

### 通俗理解

```
False（默认，打乱）：
  每个 epoch 开始时，数据顺序随机重排
  ✅ 防止模型记住数据顺序，提升泛化能力

True（不打乱）：
  数据按原始顺序喂给模型
  ⚠️ 可能导致过拟合/训练不稳定
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 几乎所有情况 | `False` ✅ | 打乱是标准做法 |
| 数据有严格时间顺序依赖 | `True` | 如时间序列预测 |
| 调试复现（想每次结果一样） | `True` | 排除随机性干扰 |

> 📌 **结论：永远保持 `False`，除非你有非常特殊的理由。**

## 11. `plot_loss`

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 是否保存训练 loss 曲线图 |

### 通俗理解

```
False：不画图
True：训练结束后生成 loss 曲线图（保存为图片文件）

曲线能告诉你：
  📉 loss 持续下降 → 正常学习
  📈 loss 先降后升 → 过拟合
  〰️ loss 震荡 → 学习率太大
  ➡️ loss 不动 → 学习率太小 / 数据有问题
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 所有训练 | **`True`** ✅ | 强烈建议开启 |
| 生产环境批量跑 | `False` | 省 I/O |

> 📌 **结论：建议永远设 `True`，这是你判断训练好坏的最直观工具。**

## 12. `include_effective_tokens_per_second`

| 项目 | 内容 |
|------|------|
| **类型** | `bool` |
| **默认值** | `False` |
| **一句话解释** | 是否计算"有效 token 吞吐量"（每秒处理多少 token） |

### 通俗理解

```
False：不统计
True：在日志中输出 tokens/s 指标

用途：
  - 对比不同 batch_size 的效率
  - 对比不同硬件的训练速度
  - 判断数据加载是否是瓶颈
```

### 什么时候改？

| 情况 | 设置 | 原因 |
|------|------|------|
| 正常训练 | `False` ✅ | 不需要 |
| 性能调优 / 对比硬件 | `True` | 量化训练效率 |
| 怀疑数据加载慢 | `True` | 看 tokens/s 是否低于预期 |

> 📌 **结论：做性能分析时设 `True`，平时不用管。**

## 总结速查表

| 参数 | 默认值 | 你需要改吗？ | 改成什么？ |
|------|:------:|:------:|------|
| `pure_bf16` | `False` | ❌ 一般不改 | — |
| `stage` | `sft` | ⚠️ 看任务 | 对齐→`dpo`，预训练→`pt` |
| `finetuning_type` | `lora` | ⚠️ 看资源 | 资源够→`full` |
| `use_llama_pro` | `False` | ❌ 一般不改 | — |
| `use_adam_mini` | `False` | ❌ 一般不改 | 超大模型全量→`True` |
| `freeze_vision_tower` | `True` | ⚠️ 多模态看情况 | 特殊视觉→`False` |
| `freeze_multi_modal_projector` | `True` | ⚠️ 多模态看情况 | 图文对齐→`False` |
| `train_mm_proj_only` | `False` | ⚠️ 多模态看情况 | 轻量适配→`True` |
| `compute_accuracy` | `False` | ✅ 调试时改 | 调试→`True` |
| `disable_shuffling` | `False` | ❌ 不要改 | — |
| `plot_loss` | `False` | ✅ **建议改** | → `True` |
| `include_effective_tokens_per_second` | `False` | ❌ 一般不改 | — |

---

> 🎯 **一句话总结：大多数参数保持默认即可。你真正需要关心的只有 `stage`、`finetuning_type`、`plot_loss` 这三个。多模态任务额外关心三个 freeze 参数。**
